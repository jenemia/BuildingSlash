# 모바일 너비 축소 + 블록 크기 조정 + 배경 이미지 구현 스펙

## 1) Goal and background
- 목표: 1080 기준 해상도 정책을 도입하고, 실제 디바이스 스크린 비율을 읽어 씬/전투/UI가 동일한 비율 규칙으로 스케일되도록 통합한다. 여기에 safe area를 반영해 노치/제스처 영역에서도 UI가 안전하게 보이게 한다.
- 배경: 현재 모바일에서 체감 폭과 요소 밀도가 기기별로 달라 보이고, 블록 크기/배경/UI 정렬이 화면마다 일관되지 않다. 따라서 `기준 해상도 + 비율 스케일 + safe area` 기반 레이아웃 체계가 필요하다.

## 2) Scope (In/Out)
### In
- 1080 기준(기준 해상도 축) 도입
- 실제 스크린 크기/비율 기반 스케일 계산 기능 구현
- 씬(월드/전투) + UI 공통 비율 적용
- safe area 반영(노치/제스처 영역 회피)
- 블록 크기 재조정(동시 가시 블록 수 4개 기준)
- 전투 씬 배경 이미지 레이어 적용
- 웹 빌드/실행 검증

### Out
- 신규 전투 시스템/밸런스 대개편
- 배경 패럴랙스/후처리 이펙트 추가
- 아트 리소스 자체 리터칭

## 3) User scenarios
1. 모바일 사용자는 전투 진입 시 화면 폭이 너무 넓지 않아 캐릭터/블록 집중이 쉬워야 한다.
2. 플레이 중 블록 폭이 화면 기준 4개 단위 체감으로 일관되어 판단이 쉬워야 한다.
3. 전투 씬 진입 시 배경 이미지가 깨짐 없이 화면을 채워 몰입감이 좋아야 한다.

## 4) Feature list with priorities
- P0: 모바일 기준 플레이 폭 축소
- P0: 블록 크기 조정(4개 가시 기준)
- P1: 배경 이미지 렌더링 적용
- P1: 웹 빌드 및 모바일 비율 검증

## 5) Data and model
- 조정 대상 파라미터(예상)
  - 기준 해상도 축: 1080 (세로 기준 정책)
  - 실화면 `screen_size`, `aspect_ratio`
  - safe area insets(`left/right/top/bottom`) 
  - 블록 폭 계산식 상수(가시폭/4 기준)
  - 배경 스케일 모드(cover 우선)
- 공통 레이아웃 모델
  - `design_rect`(기준 좌표계)와 `content_rect`(safe area 반영 좌표계) 분리
  - 월드/카메라/UI 모두 `content_rect` 기준으로 정렬
- 리소스
  - 배경 텍스처 경로(기존 art/ 또는 scenes 리소스 참조)

## 6) API/events/flow
- 앱/씬 로드 시:
  1. 실제 스크린 크기/비율 수집
  2. safe area insets 수집
  3. 1080 기준으로 스케일 팩터 및 `content_rect` 계산
  4. 월드/카메라/UI 레이아웃에 공통 적용
  5. 블록 폭 계산값 재산출
  6. 배경 이미지 노드 로드 및 화면 커버
- 리사이즈/회전 이벤트 시 동일 계산 재적용
- 별도 외부 API 없음

## 7) UI/UX
- 모바일 세로 화면에서 좌우 여백/밀도 과다를 줄이고 핵심 플레이 영역 집중
- 블록 간격/폭이 과도하게 작거나 커 보이지 않도록 균형 유지
- 배경은 UI 가독성을 해치지 않게 명도/대비 확인

## 8) Error and edge cases
- 극단적 해상도(초소형/초대형)에서 safe area가 과도하게 커질 수 있음
- 회전/리사이즈 순간 레이아웃 재계산 타이밍 이슈 가능
- 카메라/월드와 UI가 서로 다른 기준으로 계산되면 정렬 어긋남 가능
- 블록 크기 변경으로 충돌/낙하 간격 밸런스 변동 가능
- 배경 이미지 비율/해상도 부족 시 확대 블러 발생 가능

## 9) Definition of Done (DoD)
- 1080 기준 스케일 정책이 코드로 정리되고 재사용 가능
- 실제 화면 비율에 따라 씬/월드/UI가 동일 비율 규칙으로 동작
- safe area 적용 시 주요 UI가 노치/제스처 영역과 겹치지 않음
- 같은 화면에서 블록이 4개 기준으로 가시/체감됨
- 전투 배경 이미지가 씬에 표시되고 비율 왜곡/빈 영역 없음
- 웹 빌드 성공 및 실행 확인

## 10) Constraints
- 기존 코드 스타일/폴더 구조 유지
- 변경 범위 최소화(관련 씬/스크립트 위주)
- 하드코딩 수치는 주석 또는 상수화로 의미 명확화
- 불필요한 라이브러리 추가 금지

## Ticketed execution plan

### T1. 1080 기준 + 스크린 비율/세이프에어리어 레이아웃 시스템 구축
- Purpose
  - 기기별 해상도 차이에도 동일한 비율 규칙으로 월드/UI를 배치
- Changed files
  - `scripts/**` (레이아웃 계산 유틸 또는 매니저)
  - `project.godot` 또는 공통 초기화/싱글톤 설정(필요 시)
- Implementation details
  - 기준축 1080 정책 정의(세로 기준)
  - `screen_size`, `aspect_ratio`, `safe_area_insets`를 읽어 `content_rect` 계산
  - 리사이즈/회전 시 재계산 훅 추가
- Acceptance criteria
  - 디바이스/해상도 변경 시 비율 계산이 일관 동작
  - safe area 계산값을 통해 콘텐츠 안전 영역 확보
- Verification
  - 서로 다른 비율 시뮬레이션에서 `content_rect` 로그/디버그 확인

### T2. 월드/카메라 + UI 공통 비율 적용
- Purpose
  - 씬과 UI가 같은 레이아웃 기준을 공유
- Changed files
  - `scenes/**` (전투/루트 씬, UI 루트 앵커)
  - `scripts/**` (카메라 범위/오프셋, UI 배치 적용)
- Implementation details
  - 카메라 가시 영역을 `content_rect` 기준으로 정렬
  - UI 컨테이너를 safe area 안쪽으로 재배치
- Acceptance criteria
  - 월드와 UI 기준선이 어긋나지 않고 동일 비율 유지
  - 노치/제스처 영역에서 UI 침범 없음
- Verification
  - 모바일 비율(세로, 노치 가정)에서 HUD/버튼 위치 확인

### T3. 블록 4개 기준 재조정 + 배경 적용
- Purpose
  - 가시 폭 기준 블록 4개 체감 유지, 배경 화면 커버
- Changed files
  - `scripts/**` (블록 폭/스폰/충돌 계산)
  - `scenes/**` (배경 노드)
  - `art/**` (이미지 참조)
- Implementation details
  - 블록 폭 계산식을 `content_rect.width / 4` 기반으로 조정
  - 충돌체/간격 동기화
  - 배경은 cover 모드로 적용해 화면 빈틈 제거
- Acceptance criteria
  - 전투 화면에서 블록 4개 체감 달성
  - 배경이 비율 유지하며 화면을 채움
- Verification
  - 실제 플레이/미리보기에서 가시 블록 수, 충돌, 배경 커버 확인

### T4. 통합 검증 + 별도 브랜치 커밋/푸시
- Purpose
  - 변경 사항 검증 후 별도 브랜치에 커밋/푸시
- Changed files
  - 위 티켓 전체 변경 파일
- Implementation details
  - 검증 커맨드 실행(프로젝트 표준)
  - 브랜치 생성, 커밋 메시지 작성, 원격 푸시
- Acceptance criteria
  - 브랜치가 원격에 존재하고 변경 내역 확인 가능
- Verification
  - `git status`, `git log -1`, `git push` 결과 확인
