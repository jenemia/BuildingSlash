# 작업 규칙 (운영 고정)

이 문서는 BuildingSlash 작업 시 항상 적용하는 규칙이다.

## 1) 커밋 메시지 규칙
- 모든 커밋 메시지는 **한글**로 작성한다.
- 예시
  - `기능: 전투 HUD 추가`
  - `수정: 필살기 게이지 계산 오류 보정`
  - `문서: 티켓 상태 최신화`

## 2) 티켓 작업 흐름
- 티켓 단위로 작업한다.
- 각 티켓 완료 시 아래를 순서대로 수행한다.
  1. 구현 완료
  2. 웹 빌드 실행 (`./scripts/build_web.sh`)
  3. 검증
  4. 커밋 및 `main` 푸시

## 3) 배포 시 버전 증가 규칙
- 웹빌드 후 배포(푸시) 시 버전은 **0.0.1 → 0.0.2 형태로 +1** 증가한다.
- 즉, 현재 프로젝트 운영 규칙에서 배포 증가 단위는 **patch(세 번째 자리)** 로 고정한다.

## 4) 권장 배포 명령
- 버전 + 웹빌드 + 커밋 + 푸시를 한 번에 처리할 때:
  - `./scripts/release/release_web_push.sh patch "<한글 메시지>"`

## 5) 커뮤니케이션
- 메인채널에서 작업 요청을 받으면, 먼저 **새 작업 스레드**를 생성한다.
- 이후 답변/진행상황/추가 요청 처리는 해당 스레드에서 진행한다.
- 새 스레드 시작 시 이 규칙 요약을 먼저 공유하고 시작한다.
